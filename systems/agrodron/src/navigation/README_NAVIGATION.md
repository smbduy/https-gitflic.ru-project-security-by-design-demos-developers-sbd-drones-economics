### Компонент «Навигация» агродрона

Этот документ описывает, **как другим компонентам системы взаимодействовать с компонентом навигации** через брокер сообщений и (в режиме SITL) через SITL‑адаптер.

Формат имён топиков для всех компонентов дрона:

- `agrodron.<имя_компонента>.<подсистема>`

Для навигации используется префикс `agrodron.navigation.*`.

---

### 1. Назначение компонента

- Принимать **сырые навигационные данные** (NMEA‑0183 / сообщения SITL‑адаптера).
- Выполнять **валидацию и преобразование** этих данных в единый JSON‑формат навигационного состояния.
- Публиковать текущее навигационное решение в топик:
  - `agrodron.navigation.state`
- Предоставлять **единый источник истины по положению дрона** для:
  - автопилота;
  - ограничителя;
  - компонента экстренных ситуаций (`emergensy`);
  - телеметрии и мониторинга.

Компонент навигации **не принимает решений по управлению полётом** и **не инициирует аварийные сценарии** — он только поставляет корректные и актуальные координаты и качество навигации.

---

### 2. Используемые форматы данных

#### 2.1. Архитектура SITL и взаимодействие навигации

По документу «Алгоритм работы SITL-адаптера»:

- **Внешний SITL-адаптер** — отдельный контейнер/сервис, по запросу по шине возвращает JSON с навигационными данными (lat, lon, alt, vx, vy, heading и т.д.).

- **Компонент `sitl_adapter`** (или иной SITL-сервис) — подписан на топик `SITL_TOPIC`, по action `get_state` отвечает JSON (сырое состояние: sitl.position.v1, Redis-формат, NMEA-derived и т.п.).

- **Навигация** — опрашивает SITL только через шину:
  - `proxy_request` → `components.security_monitor` → `target: SITL_TOPIC`, `action: get_state`
  - В ответ получает JSON, нормализует его (sitl_normalizer) в единый NAV_STATE.
  - Redis не используется; взаимодействие только request/response по шине.

Поддерживаемые входные форматы:

**sitl.position.v1** (MQTT):

```json
{"drone_id":"drone_001","lat":59.9386,"lon":30.3165,"alt":100.2,"vx":1.23}
```

**Redis-state** (drone:{id}:state):

```json
{"lat":59.938623,"lon":30.316534,"alt":100.0,"vx":1.23,"vy":0.87,"vz":0.0,"heading":25.8,"status":"MOVING","last_update":"2026-03-08T16:40:00Z"}
```

**NMEA-derived** (legacy):

```json
{
  "derived": {"lat_decimal": 60.123450, "lon_decimal": 30.123400, "altitude_msl": 4.9},
  "nmea": {"rmc": {"speed_knots": 9.3, "course_degrees": 90.0}, "gga": {"quality": 1, "satellites": 14, "hdop": 0.7}}
}
```

#### 2.2. Выходной формат навигационного состояния

Компонент навигации публикует упрощённый JSON‑формат, уже согласованный с автопилотом, ограничителем и `emergensy`:

```json
{
  "action": "NAV_STATE",
  "timestamp": "2026-03-09T12:00:01.234Z",
  "lat": 60.123450,
  "lon": 30.123400,
  "alt_m": 4.9,
  "ground_speed_mps": 4.8,
  "heading_deg": 90.0,
  "fix": "3D",        // NONE | 2D | 3D
  "satellites": 14,
  "hdop": 0.7
}
```

Пояснения:

- **`lat`, `lon`** — координаты в градусах (WGS‑84), полученные из `derived.lat_decimal` / `derived.lon_decimal` или распарсенных NMEA.
- **`alt_m`** — высота в метрах (над уровнем моря или относительно HOME — определяется политикой системы; для SITL можно использовать `altitude_msl`).
- **`ground_speed_mps`** — путевая скорость по земле, в м/с (из `speed_knots` с пересчётом).
- **`heading_deg`** — курс (0–359.9°), из `course_degrees` или вычисленный по векторам скорости.
- **`fix`** — тип решения: `NONE` / `2D` / `3D` (на основе полей `quality`, `gps_valid`, `satellites_used`).
- **`satellites`** — количество спутников в решении.
- **`hdop`** — горизонтальная точность.

Дополнительно навигация добавляет служебные поля:

- `gps_valid` — булев признак достаточного качества GNSS (например, `fix = "3D"` и `satellites >= 4`);
- `drone_id` — идентификатор дрона (`drone_001` и т.п.; может приходить от SITL или подставляться из конфигурации).

Частота публикации — **5–20 Гц** (настраивается внутри компонента навигации и должна быть согласована с автопилотом и ограничителем). В текущем прототипе используется 10 Гц.

---

### 3. Топики навигации

#### 3.1. Входящие топики (на которые подписывается навигация)

1. **Данные от SITL‑адаптера / внешнего источника**

   - **Внутренний топик компонента SITL‑адаптера**: `components.sitl_adapter`
   - **Кто пишет**: сам SITL‑адаптер (или обёртка над реальным GNSS/симулятором).
   - **Кто читает**: монитор безопасности.
   - **Назначение**: предоставить по запросу актуальное навигационное состояние для конкретного дрона.

   Навигация не использует прямую подписку на `sitl/telemetry`, а всегда ходит к адаптеру через монитор безопасности.

2. **Конфигурация навигации (опционально)**

   - **Топик**: `agrodron.navigation.config`
   - **Кто пишет**: конфигурационный сервис / оператор.
   - **Назначение**: изменение частоты публикации, порогов качества навигации, фильтров.

   Пример сообщения:

   ```json
   {
     "min_satellites": 6,
     "max_hdop": 1.5,
     "publish_rate_hz": 10
   }
   ```

   Если качество сигнала ниже порогов (`satellites < min_satellites` или `hdop > max_hdop`), навигация может понизить `fix` до `NONE`/`2D` и/или пометить состояние как деградированное (см. раздел о логике).

#### 3.2. Исходящие топики (которые заполняет навигация)

1. **Состояние навигации (основной поток)**

   - **Топик**: `agrodron.navigation.state`
   - **Кто пишет**: `navigation`.
   - **Кто читает**:
     - автопилот (`agrodron.autopilot`) — использует для расчёта движения по миссии;
     - ограничитель (`agrodron.limiter`) — анализ отклонений от маршрута;
     - компонент экстренных ситуаций (`emergensy`) — фиксация факта посадки и начальной позиции аварийного протокола;
     - телеметрия/мониторинг — отображение текущего положения.

   Формат сообщения см. в разделе 2.2.

2. **Статус компонента навигации (опционально)**

   - **Топик**: `agrodron.navigation.state_internal`
   - **Кто пишет**: `navigation`.
   - **Кто читает**: телеметрия / мониторинг.
   - **Назначение**: дать представление о качестве навигационного решения.

   Пример:

   ```json
   {
     "timestamp": "2026-03-09T12:00:01.240Z",
     "state": "OK",             // OK | DEGRADED | NO_FIX | ERROR
     "fix": "3D",
     "satellites": 14,
     "hdop": 0.7,
     "source": "SITL",          // SITL | GNSS | OTHER
     "last_update_age_ms": 50
   }
   ```

### 4. Логика работы навигации (в общих чертах)

1. **Инициализация**

   - Подключиться к брокеру сообщений.
   - Загрузить начальную конфигурацию (`min_satellites`, `max_hdop`, `publish_rate_hz`, `drone_id`).

2. **Периодический опрос SITL‑адаптера через монитор**

- С периодом, соответствующим `publish_rate_hz` (например, 10 Гц), навигация:
  - формирует запрос в монитор безопасности (`proxy_request` на топик SITL с action `get_state`);
  - получает от адаптера NAV_STATE‑подобный JSON.

3. **Приём и нормализация данных**

- Получить очередное сообщение от SITL‑адаптера (в ответ на запрос) или GNSS.
   - Выполнить базовую валидацию:
     - корректность типов полей;
     - диапазоны координат и высоты;
     - валидность временной метки.
   - Вычислить/извлечь:
     - широту/долготу в градусах (`lat`, `lon`);
     - высоту в метрах (`alt_m`);
     - путевую скорость в м/с (`ground_speed_mps`);
     - курс (`heading_deg`);
     - показатели качества (`fix`, `satellites`, `hdop`).

4. **Оценка качества навигации**

   - На основе `quality` / `gps_valid` / `satellites_used` / `hdop` определить:
     - `fix = "3D"` — если спутников ≥ порога и `hdop` в допустимых границах;
     - `fix = "2D"` — если координаты есть, но качество ниже;
     - `fix = "NONE"` — если данных недостаточно или они невалидны.
   - Если качество ниже порогов конфигурации:
     - перевести внутренний статус в `DEGRADED` или `NO_FIX`;
     - по необходимости понизить частоту обновлений.

5. **Публикация навигационного состояния**

   - Сформировать JSON согласно разделу 2.2.
   - Периодически (с частотой `publish_rate_hz`, например 10 Гц) публиковать сообщения в топик `agrodron.navigation.state`.
   - Обновлять служебный статус в `agrodron.navigation.state_internal`.

6. **Поведение при ошибках**

   - При критической ошибке входных данных (например, скачок координат на километры) навигация:
     - может игнорировать «подозрительное» сообщение и продолжить использовать последнее корректное состояние;
     - помечает своё состояние как `ERROR`/`DEGRADED` в служебном топике.
   - Навигация **не принимает решений по посадке** — она только даёт качество/координаты, на основе которых реагируют `limiter` и `emergensy`.

---

### 5. Кратко: кто с чем работает (связи с другими компонентами)

- **SITL‑адаптер**:
  - Представлен как компонент `components.sitl_adapter`.
  - Получает запросы через монитор безопасности, возвращает NAV_STATE‑подобный JSON.
  - Не общается с навигацией напрямую, только через монитор.

- **Навигация**:
  - Пишет в `agrodron.navigation.state` — единый поток навигационных данных.
  - Опционально пишет в `agrodron.navigation.state_internal` — служебное состояние и качество.

- **Автопилот**:
  - Подписывается на `agrodron.navigation.state`.
  - Использует `lat`, `lon`, `alt_m`, `ground_speed_mps`, `heading_deg`, `fix`, `satellites`, `hdop` для выполнения миссии.

- **Ограничитель (`limiter`)**:
  - Подписывается на `agrodron.navigation.state`.
  - На основе координат и качества навигации проверяет отклонение от маршрута и пороги безопасности, генерирует события в `agrodron.limiter.event`.

- **Компонент экстренных ситуаций (`emergensy`)**:
  - Подписывается на `agrodron.navigation.state`.
  - Берёт последнюю известную позицию для формирования событий `EMERGENCY_PROTOCOL_STARTED` и фиксации места аварийной посадки.

- **Телеметрия / мониторинг**:
  - Подписываются на `agrodron.navigation.state` и `agrodron.navigation.state_internal` для отображения координат, курса, качества GNSS.

---

### 6. Архитектура SITL (обновлённая)

Навигация запрашивает данные SITL по шине:

- **Навигация** с периодом 10 Гц отправляет в монитор безопасности `proxy_request` с `target: { topic: SITL_TOPIC, action: get_state }`, в `data` может передаваться `drone_id`.
- **SITL-адаптер** (или любой сервис на топике `SITL_TOPIC`) по action `get_state` возвращает JSON (сырые поля: lat, lon, alt, vx, vy, heading и т.д.).
- Ответ приходит в поле `target_response` (или `target_response.payload`). Навигация нормализует его в NAV_STATE (см. `sitl_normalizer.py`).

---

### 7. Безопасность и конфигурация (по шаблону ТЗ)

- **Проверка отправителя**: входящие сообщения принимаются **только от МБ** (`sender.startswith("security_monitor")`).
- **Конфигурация (env-driven)**: `config.py`, `.env.example` — `SYSTEM_NAME`, `SITL_TOPIC`, `SITL_GET_STATE_ACTION`, `NAVIGATION_POLL_INTERVAL_S`, `NAVIGATION_REQUEST_TIMEOUT_S` и др.
- **Структура компонента**:
  ```
  navigation/
  ├── config.py
  ├── src/navigation.py
  ├── src/sitl_normalizer.py
  ├── topics.py
  ├── __main__.py
  ├── .env.example
  ├── docker/Dockerfile
  └── tests/
  ```
