### Компонент «Журнал» агродрона

Этот документ описывает, **как другим компонентам системы взаимодействовать с компонентом журнала** (логирования) через брокер сообщений.

Формат имён топиков для всех компонентов дрона:

- `agrodron.<имя_компонента>.<подсистема>`

Для журнала используется:

- внутренний топик компонента: `components.journal`;
- при необходимости можно добавить внешний поток `agrodron.journal.event` (например, для аналитики или внешнего мониторинга).

---

### 1. Назначение компонента

- Принимать **события и служебные записи** от других компонентов (через монитор безопасности).
- Сохранять каждую запись в **неизменяемый файл журнала** (append-only, формат JSON‑строка на линию).
- Обеспечивать **единое место хранения** критических событий жизненного цикла дрона:
  - аварийные события от ограничителя / `emergensy`;
  - ключевые шаги миссии от автопилота (по мере развития системы);
  - сообщения от телеметрии / мониторинга.

Компонент журнала **ничего не решает сам** (не влияет на управление дроном), он только аккумулирует информацию для последующего анализа и расследований.

---

### 2. Формат данных и место хранения

#### 2.1. Файл журнала

По умолчанию журнал пишет в файл:

- `/data/agrodron_journal.ndjson`

Путь настраивается через переменную окружения:

- `JOURNAL_FILE_PATH=/data/agrodron_journal.ndjson`

Формат файла:

- **NDJSON** (Newline Delimited JSON) — по одной JSON‑записи в строке;
- записи **никогда не изменяются и не удаляются**, возможна только дописывающая запись (append-only);
- ротация/архивация файлов при необходимости выполняется внешними средствами (logrotate, sidecar и т.п.).

Каждая запись имеет общий каркас:

```json
{
  "timestamp": "2026-03-09T12:00:03.050Z",
  "source_component": "emergensy",
  "source_action": "LOG_EVENT",
  "level": "INFO",
  "event": "EMERGENCY_PROTOCOL_STARTED",
  "payload": {
    "...": "оригинальные поля события"
  }
}
```

---

### 3. Топики журнала

#### 3.1. Входящий топик (на который подписывается журнал)

1. **Внутренний топик компонента журнала**

   - **Топик**: `components.journal`
   - **Кто пишет**: другие компоненты **через монитор безопасности**:
     - `emergensy`;
     - `limiter`;
     - `autopilot`;
     - телеметрия и др.
   - **Назначение**: приём структурированных событий для записи в файл.

   Типовой запрос от компонента (перед отправкой в монитор):

   ```json
   {
     "action": "proxy_publish",
     "sender": "emergensy",
     "payload": {
       "target": {
         "topic": "components.journal",
         "action": "LOG_EVENT"
       },
       "data": {
         "event": "EMERGENCY_PROTOCOL_STARTED",
         "mission_id": "mission-1",
         "details": {
           "distance_from_path_m": 15.0,
           "max_distance_from_path_m": 10.0
         }
       }
     }
   }
   ```

   Монитор безопасности по своей логике проксирует это сообщение к `components.journal`,
   и журнал получает уже нормализованный вид:

   ```json
   {
     "action": "LOG_EVENT",
     "sender": "security_monitor...",
     "payload": {
       "event": "EMERGENCY_PROTOCOL_STARTED",
       "mission_id": "mission-1",
       "details": {
         "distance_from_path_m": 15.0,
         "max_distance_from_path_m": 10.0
       }
     }
   }
   ```

---

### 4. Поддерживаемые actions

Компонент журнала обрабатывает одно действие (на своём топике `components.journal`):

1. **`LOG_EVENT`**

   - **Назначение**: записать одну запись журнала.
   - **Формат сообщения (на топике `components.journal`)**:

   ```json
   {
     "action": "LOG_EVENT",
     "sender": "security_monitor_emergensy",
     "payload": {
       "event": "EMERGENCY_PROTOCOL_STARTED",
       "mission_id": "mission-1",
       "details": {
         "distance_from_path_m": 15.0,
         "max_distance_from_path_m": 10.0
       }
     }
   }
   ```

   - **Поведение журнала**:
     - формирует запись с полями:
       - `timestamp` — время обработки;
       - `source_component` — логическое имя источника (из `payload.source` или по умолчанию из `sender`);
       - `event` — тип события (`payload.event`);
       - `payload` — весь `payload` сообщения;
     - дописывает JSON‑строку в файл `JOURNAL_FILE_PATH`.

---

### 5. Логика работы журнала (в общих чертах)

1. **Инициализация**

   - Подключиться к брокеру сообщений.
   - Прочитать переменную окружения `JOURNAL_FILE_PATH` (или использовать значение по умолчанию `/data/agrodron_journal.ndjson`).
   - Убедиться, что директория для файла существует (создать при необходимости).
   - Подписаться на топик `components.journal`.

2. **Приём событий (`LOG_EVENT`)**

   - Для каждого входящего сообщения:
     - взять `payload` как тело события;
     - определить `source_component`:
       - сначала `payload.source` (если есть),
       - иначе — логическое имя из `sender`;
     - сформировать объект журнала (см. раздел 2.1);
     - сериализовать в JSON и дописать строку в файл (`append`).

3. **Непрерывность и неизменяемость записи**

   - Журнал **не изменяет и не удаляет** уже записанные строки.
   - При ошибке записи (например, недоступна файловая система) журнал:
     - логирует ошибку в stdout/stderr;
     - может продолжить попытки записи следующих событий, когда файловая система восстановится.

4. **Взаимодействие с другими компонентами**

   - `emergensy`:
     - отправляет ключевые события аварийного протокола через монитор безопасности на `components.journal` (`LOG_EVENT`).
   - `limiter`:
     - может отправлять события превышения порогов в журнал (по мере развития системы).
   - `autopilot`, `navigation`, телеметрия:
     - могут слать в журнал свои собственные события (изменения состояния, диагностика) тем же способом.

---

### 6. Безопасность и конфигурация (по шаблону ТЗ)

- **Проверка отправителя**: входящие сообщения принимаются **только от МБ** (`sender.startswith("security_monitor")`). Сообщения от других отправителей молча игнорируются.
- **Конфигурация (env-driven)**: `config.py`, `.env.example`:
  - `SYSTEM_NAME` — имя системы (по умолчанию `components`);
  - `COMPONENT_TOPIC` — топик компонента (по умолчанию `{SYSTEM_NAME}.journal`);
  - `JOURNAL_FILE_PATH` — путь к NDJSON-файлу журнала.
