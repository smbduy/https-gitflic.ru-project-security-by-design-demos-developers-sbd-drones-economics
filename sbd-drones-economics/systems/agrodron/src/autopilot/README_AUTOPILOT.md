## Компонент «Автопилот» (`autopilot`) системы agrodron

Этот документ описывает, **как взаимодействовать с автопилотом** в архитектуре:

- у каждого компонента **ровно один** входящий топик `agrodron.<component>`;
- **любой** входящий запрос к компоненту приходит **только от МБ** (`security_monitor`);
- компоненты **не подписываются** на чужие топики;
- данные между компонентами получаются только через **`proxy_request` (polling)** через МБ.

---

## 1. Назначение компонента

- Принимает **подготовленную миссию** от обработчика миссий.
- Поддерживает **жизненный цикл миссии**: загрузка, запуск, пауза, возобновление, завершение, отмена.
- Выполняет миссию, **используя навигационные данные**, получаемые запросом в компонент `navigation` через МБ.
- Управляет приводами и опрыскивателем через МБ (команды в `agrodron.motors` и `agrodron.sprayer`).
- Отдаёт **состояние миссии и автопилота** по запросу через МБ.
- **Не обрабатывает аварийные ситуации**: аварийные сценарии реализуются отдельными компонентами/политиками.

---

## 2. Топик и actions автопилота

### 2.1. Входящий топик

- **Топик**: `agrodron.autopilot`
- **Кто пишет**: только МБ (`agrodron.security_monitor`) при проксировании.
- **Кто читает**: автопилот.

### 2.2. Поддерживаемые actions

#### 2.2.1. `mission_load` — загрузка миссии

Сообщение на топик `agrodron.autopilot` (приходит от МБ):

```json
{
  "action": "mission_load",
  "sender": "security_monitor_...",
  "payload": {
    "mission": {
      "mission_id": "string",
      "steps": [
        { "id": "wp-001", "lat": 60.123456, "lon": 30.123456, "alt_m": 5.0, "speed_mps": 5.0, "spray": true }
      ]
    }
  }
}
```

Отправитель миссии (например `mission_handler`) вызывает это **через МБ**:

```json
{
  "action": "proxy_request",
  "sender": "mission_handler",
  "payload": {
    "target": { "topic": "agrodron.autopilot", "action": "mission_load" },
    "data": { "mission": { "...": "..." } }
  }
}
```

#### 2.2.2. `cmd` — команды управления

Команды также доставляются через МБ на `agrodron.autopilot`:

```json
{
  "action": "cmd",
  "sender": "security_monitor_...",
  "payload": {
    "command": "START"  // START | PAUSE | RESUME | ABORT | RESET | EMERGENCY_STOP | KOVER
  }
}
```

Семантика команд:

- `START` – начать выполнение загруженной миссии.
- `PAUSE` – приостановить выполнение (дрон переходит в безопасный статичный режим, опрыскиватель выключен).
- `RESUME` – продолжить с текущего шага, если всё в норме.
- `ABORT` – штатно прервать миссию, перейти в состояние «отменена».
- `RESET` – сбросить текущее состояние автопилота в «IDLE` (без аварийной логики).
- `EMERGENCY_STOP` – перевести автопилот в состояние `EMERGENCY_STOP`, он перестаёт отдавать команды в `actuators`.
- `KOVER` – инициировать посадку до земли и перейти в ожидание (реализация прототипа).

#### 2.2.3. `get_state` — snapshot состояния автопилота

Запрос делается через МБ (`proxy_request`) на `agrodron.autopilot` с `action="get_state"`.

---

## 3. Взаимодействия автопилота с другими компонентами (через МБ)

### 3.1. Получение навигации (polling через `proxy_request`)

Автопилот **не подписывается** на навигацию. Он раз в \(N\) секунд делает `proxy_request` в `agrodron.navigation`:

```json
{
  "action": "proxy_request",
  "sender": "autopilot",
  "payload": {
    "target": { "topic": "agrodron.navigation", "action": "get_state" },
    "data": {}
  }
}
```

Частота polling настраивается через `.env`:

- `AUTOPILOT_NAV_POLL_INTERVAL_S`
- `AUTOPILOT_REQUEST_TIMEOUT_S`

Action навигации настраивается:

- `NAVIGATION_GET_STATE_ACTION` (по умолчанию `get_state`)

### 3.2. Управление приводами и опрыскивателем (через `proxy_publish`)

Автопилот отправляет команды через МБ:

 - `agrodron.motors` / `SET_TARGET`
 - `agrodron.sprayer` / `SET_SPRAY`

---

## 4. Параметры `.env` (основные)

См. `components/autopilot/.env.example`. Ключевые параметры:

 - `SYSTEM_NAME`
 - `AUTOPILOT_CONTROL_INTERVAL_S`
 - `AUTOPILOT_NAV_POLL_INTERVAL_S`
 - `AUTOPILOT_REQUEST_TIMEOUT_S`

---

## 5. Формат миссии (JSON)

Автопилот работает с **упрощённым форматом** миссии. Основная цель – удобство реализации и тестирования в SITL.

Пример миссии:

```json
{
  "mission_id": "mission-1",
  "home": {
    "lat": 60.000000,
    "lon": 30.000000,
    "alt_m": 0.0
  },
  "steps": [
    {
      "id": "wp-001",
      "type": "WAYPOINT",      // пока достаточно WAYPOINT
      "lat": 60.123456,
      "lon": 30.123456,
      "alt_m": 5.0,
      "speed_mps": 5.0,
      "spray": false
    },
    {
      "id": "wp-002",
      "type": "WAYPOINT",
      "lat": 60.123500,
      "lon": 30.123800,
      "alt_m": 5.0,
      "speed_mps": 5.0,
      "spray": true
    }
  ]
}
```

Минимально обязательные поля для шага:

- `id` – строковый идентификатор шага.
- `type` – пока достаточно `WAYPOINT`.
- `lat`, `lon` – координаты в градусах (WGS84).
- `alt_m` – высота в метрах над уровнем моря или относительно home (решается на стороне навигации/СИТЛ).
- `speed_mps` – желаемая скорость.
- `spray` – флаг включения опрыскивателя **на отрезке до следующей точки**.

Все дополнительные «красивые» фичи (типы точек, длительность зависания и т.п.) можно добавить позже.
