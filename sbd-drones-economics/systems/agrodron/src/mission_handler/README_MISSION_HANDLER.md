### Компонент «Обработчик миссий» агродрона

Этот документ описывает, **как другим компонентам системы взаимодействовать с обработчиком миссий** через брокер сообщений.

Формат имён топиков для всех компонентов дрона:

- `agrodron.<имя_компонента>.<подсистема>`

Для обработчика миссий используется:

- внутренний топик компонента: `components.mission_handler` (или `agrodron.mission_handler` при `SYSTEM_NAME=agrodron`);
- топики строятся динамически через `config.py` и `SYSTEM_NAME` (см. раздел 5).

---

### 1. Назначение компонента

- Принимать **строго файлы WPL** (QGC WPL / ArduPilot Waypoint). Другие форматы на вход не принимаются — предварительное преобразование в WPL выполняется до отправки.
- Преобразовывать WPL в **JSON‑формат автопилота** (см. README автопилота).
- Через монитор безопасности инициировать загрузку миссии в автопилот:
  - отправлять `mission_load` на `components.autopilot` через `components.security_monitor`.
- Логировать ключевые события работы (получена миссия, успешно передана в автопилот, ошибка валидации и т.п.) в компонент `journal`.

Компонент **не выполняет сам полёт** и не управляет приводами — он только подготавливает и передаёт миссии в автопилот.

---

### 2. Форматы данных

#### 2.1. Входной формат — строго WPL

На вход поступают **только файлы WPL** (текстовый формат QGC WPL / ArduPilot Waypoint). Иные форматы не принимаются; преобразование в WPL должно выполняться до отправки (например, на стороне планировщика, оператора или внешней надсистемы).

Топик: `components.mission_handler`, action: `LOAD_MISSION`.

Пример сообщения (через монитор безопасности):

```json
{
  "action": "LOAD_MISSION",
  "sender": "security_monitor_operator",
  "payload": {
    "wpl_content": "QGC WPL 110\n0\t1\t0\t16\t0\t0\t0\t0\t60.000000\t30.000000\t0.0\t1\n1\t0\t3\t16\t0\t0\t0\t0\t60.123456\t30.123456\t5.0\t1",
    "mission_id": "mission-1"
  }
}
```

Поля `payload`:

- **`wpl_content`** (обязательно) — сырое содержимое WPL-файла (строка).
- **`mission_id`** (опционально) — ID миссии; если не задан, генерируется автоматически (`wpl-<timestamp>`).

Формат WPL: первая строка — заголовок (`QGC WPL 110` и т.п.), далее — строки с табуляцией между колонками: Index, Current, Frame, Command, P1–P4, Lat, Lon, Alt, Autocontinue. Поддерживаются NAV_WAYPOINT (MAV_CMD 16). Строка с index `-1` — HOME.

#### 2.2. Формат миссии для автопилота

Формат идентичен описанному в README автопилота (раздел 3), обёрнутому в команду `MISSION_LOAD`:

```json
{
  "action": "mission_load",
  "sender": "mission_handler",
  "payload": {
    "mission": {
      "mission_id": "mission-1",
      "home": { "lat": 60.0, "lon": 30.0, "alt_m": 0.0 },
      "steps": [
        {
          "id": "wp-001",
          "type": "WAYPOINT",
          "lat": 60.123456,
          "lon": 30.123456,
          "alt_m": 5.0,
          "speed_mps": 5.0,
          "spray": false
        }
      ]
    }
  }
}
```

Обработчик миссий:

- валидирует наличие обязательных полей (`mission_id`, `steps`, координаты);
- при необходимости проставляет значения по умолчанию (`type = "WAYPOINT"`, `speed_mps`, `spray`).

### 3. Топики обработчика миссий

#### 3.1. Входящие топики

1. **Внутренний топик компонента**

   - **Топик**: `components.mission_handler`
   - **Actions**:
     - `LOAD_MISSION` — загрузка и передача миссии в автопилот;
     - `VALIDATE_ONLY` — только валидация миссии без отправки в автопилот;
     - `get_state` — запрос состояния обработчика (последняя миссия, ошибка).

   Пример `LOAD_MISSION` уже показан выше.

#### 3.2. Исходящие взаимодействия

1. **Загрузка миссии в автопилот**

   - **Через монитор безопасности**:
     - Топик монитора: `components.security_monitor`
     - Action: `proxy_request` или `proxy_publish`.
   - Целевой компонент: `components.autopilot`
   - Целевое действие: `mission_load`

   Пример сообщения в монитор:

   ```json
   {
     "action": "proxy_request",
     "sender": "mission_handler",
     "payload": {
       "target": {
         "topic": "components.autopilot",
         "action": "mission_load"
       },
       "data": {
         "mission": { "...": "см. формат выше" }
       }
     }
   }
   ```

   Ответ автопилота (через монитор) обрабатывается обработчиком миссий и при необходимости логируется.

2. **Запись событий в журнал**

   - **Топик монитора безопасности**: `components.security_monitor`
   - Цель: `components.journal`, действие `LOG_EVENT`.

   Примеры событий:

   - `MISSION_HANDLER_MISSION_RECEIVED` — миссия получена и прошла базовую валидацию.
   - `MISSION_HANDLER_MISSION_SENT_TO_AUTOPILOT` — миссия успешно передана в автопилот.
   - `MISSION_HANDLER_VALIDATION_ERROR` — ошибка валидации входной миссии.
   - `MISSION_HANDLER_AUTOPILOT_ERROR` — автопилот вернул ошибку при загрузке.

### 4. Логика работы (в общих чертах)

1. **Инициализация**

   - Подключиться к брокеру сообщений.
   - Подписаться на `components.mission_handler`.
   - Подготовить структуру для хранения последней миссии и последней ошибки.

2. **Обработка `LOAD_MISSION`**

   - Принять входное сообщение; ожидается `payload.wpl_content` (строка).
   - Распарсить WPL через парсер; при ошибке парсинга — вернуть `invalid_input_wpl_required` или конкретную ошибку.
   - Преобразовать в JSON‑формат автопилота.
   - Провести базовую валидацию:
     - `mission_id` — строка;
     - `steps` — непустой список;
     - для каждого шага — координаты и высота.
   - При ошибке:
     - сохранить описание ошибки во внутреннем состоянии;
     - отправить событие `MISSION_HANDLER_VALIDATION_ERROR` в журнал;
     - вернуть ответ с `ok=false`.
   - При успешной валидации:
     - сохранить последнюю миссию во внутреннем состоянии;
     - отправить в журнал `MISSION_HANDLER_MISSION_RECEIVED`;
     - сформировать запрос к автопилоту через монитор (`proxy_request` → `components.autopilot`, `mission_load`);
     - обработать ответ:
       - при `ok=true` — отправить в журнал `MISSION_HANDLER_MISSION_SENT_TO_AUTOPILOT`;
       - при ошибке — журнал `MISSION_HANDLER_AUTOPILOT_ERROR` с деталями.

3. **Обработка `VALIDATE_ONLY`**

   - Выполнить те же проверки, что и для `LOAD_MISSION`, но **не отправлять** миссию в автопилот.
   - Вернуть результат валидации, при необходимости залогировать `MISSION_HANDLER_VALIDATION_ERROR`.

4. **Взаимодействие с другими компонентами**

   - `autopilot`:
     - получает миссии, подготовленные обработчиком миссий (через монитор).
   - `journal`:
     - получает все ключевые события жизненного цикла миссий.
   - внешние источники (операторский интерфейс, наземная станция, надсистемы):
     - отправляют сообщения `LOAD_MISSION` или `VALIDATE_ONLY` в обработчик миссий (через монитор безопасности).

---

### 5. Безопасность и конфигурация (по шаблону README_TZ_COMPONENTS)

**Проверка доверенного отправителя**

Любое входящее сообщение принимается **только от монитора безопасности** (`sender` начинается с `security_monitor`). Сообщения от других отправителей молча игнорируются (возврат `None`).

**Конфигурация (env-driven)**

Все параметры задаются через переменные окружения (см. `.env.example` и `config.py`):

- `SYSTEM_NAME` — имя системы (по умолчанию `components`);
- `COMPONENT_TOPIC` — топик компонента (по умолчанию `{SYSTEM_NAME}.mission_handler`);
- `SECURITY_MONITOR_TOPIC` — топик монитора безопасности;
- `AUTOPILOT_TOPIC` — топик автопилота;
- `JOURNAL_TOPIC` — топик журнала;
- `MISSION_HANDLER_REQUEST_TIMEOUT_S` — таймаут запросов через МБ (по умолчанию 10.0 с).
