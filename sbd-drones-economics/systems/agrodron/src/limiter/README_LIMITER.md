## Компонент «Ограничитель» агродрона

Этот документ описывает, **как другим компонентам системы взаимодействовать с ограничителем** через брокер сообщений.

Актуальная архитектура:

- каждый компонент имеет **1 входящий топик** `agrodron.<component>`;
- любой вход в компонент приходит **только от МБ** (`security_monitor`);
- компоненты **не подписываются на чужие топики**;
- данные берутся только через **`proxy_request` (polling)** через МБ.

---

## 1. Назначение компонента

- Контролировать, **не нарушает ли текущая миссия заданные ограничения** (отклонение от маршрута, нарушенные зоны и т.п.).
- Анализировать:
  - текущие навигационные данные;
  - состояние моторов и опрыскивателя;
  - информацию о текущей миссии и её шагах.
- При обнаружении отклонений выше допустимых порогов:
  - **запускать протокол аварийной посадки**, формируя соответствующее событие;
  - дальше работу выполняет отдельный компонент (например, аварийный контроллер/экстренная ситуация).

Сам ограничитель **не управляет приводами и опрыскивателем напрямую**, он только:

- наблюдает за выполнением миссии;
- инициирует аварийный сценарий, отправляя событие в `emergensy` через МБ.

---

## 2. Используемые форматы данных

Ограничитель использует те же форматы, что и автопилот.

### 2.1. Формат миссии

Формат миссии идентичен формату, который получает автопилот:

```json
{
  "mission_id": "mission-1",
  "home": {
    "lat": 60.000000,
    "lon": 30.000000,
    "alt_m": 0.0
  },
  "steps": [
    {
      "id": "wp-001",
      "type": "WAYPOINT",
      "lat": 60.123456,
      "lon": 30.123456,
      "alt_m": 5.0,
      "speed_mps": 5.0,
      "spray": false
    },
    {
      "id": "wp-002",
      "type": "WAYPOINT",
      "lat": 60.123500,
      "lon": 30.123800,
      "alt_m": 5.0,
      "speed_mps": 5.0,
      "spray": true
    }
  ]
}
```

Ограничитель получает миссию как команду `mission_load` на свой топик `agrodron.limiter` (доставляется через МБ).

### 2.2. Формат навигационных данных

Ограничитель использует JSON, формируемый навигацией:

```json
{
  "action": "NAV_STATE",
  "timestamp": "2026-03-09T12:00:01.234Z",
  "lat": 60.123450,
  "lon": 30.123400,
  "alt_m": 4.9,
  "ground_speed_mps": 4.8,
  "heading_deg": 90.0,
  "fix": "3D",
  "satellites": 14,
  "hdop": 0.7
}
```

Ограничитель **не подписывается** на навигацию. Он периодически запрашивает snapshot у `navigation` через МБ.

### 2.3. Состояние моторов и опрыскивателя

Ограничитель **не взаимодействует напрямую** с моторами и опрыскивателем и не подписывается на их внутренние топики.

Для анализа он использует **агрегированное состояние дрона**, которое предоставляет компонент телеметрии.

Рекомендуется единый JSON-формат snapshot, который отдаёт компонент `telemetry` по запросу (например action `get_state` на `agrodron.telemetry`).

Пример агрегированного состояния:

```json
{
  "timestamp": "2026-03-09T12:00:01.260Z",
  "mission_id": "mission-1",
  "autopilot_state": "EXECUTING",
  "motors": {
    "mode": "AUTO",
    "target": {
      "heading_deg": 90.0,
      "ground_speed_mps": 5.0,
      "alt_m": 5.0
    }
  },
  "sprayer": {
    "spray": true,
    "rate_lpm": 2.0
  }
}
```

Именно эти данные (через телеметрию) ограничитель использует вместе с навигацией и миссией для принятия решения о нарушении ограничений.

## 3. Топик и actions ограничителя

### 3.1. Входящий топик

1. **Внутренний топик компонента**

   - Топик: `agrodron.limiter`
   - Кто пишет: только МБ при проксировании.
   - Кто читает: limiter.
   - Actions:
     - `mission_load` — загрузка миссии
     - `update_config` — обновление порогов (опционально)
     - `get_state` — snapshot состояния limiter

### 3.2. Взаимодействия limiter с другими компонентами (через МБ)

Limiter работает в режиме polling:

1) **Навигация**: `proxy_request` в `agrodron.navigation` (action настраивается env).
2) **Телеметрия**: `proxy_request` в `agrodron.telemetry` (action настраивается env).
3) **Экстренные ситуации**: при аварии limiter делает `proxy_publish` в `agrodron.emergensy` с `action="limiter_event"` и payload вида:

```json
{
  "event": "EMERGENCY_LAND_REQUIRED",
  "details": {
    "distance_from_path_m": 15.0,
    "max_distance_from_path_m": 10.0,
    "alt_deviation_m": 5.0,
    "max_alt_deviation_m": 3.0
  }
}
```

## 4. Логика работы ограничителя (в общих чертах)

1. **Получение миссии**
   - Ограничитель получает и сохраняет миссию.
   - Строит внутреннее представление траектории (последовательность точек).

2. **Наблюдение за полётом**
   - В цикле polling:
     - запрашивает snapshot навигации;
     - (опционально) запрашивает snapshot телеметрии;
     - вычисляет отклонения.
   - Сравнивает расстояние и высоту с допустимыми порогами.

3. **Пороги и реакции**
   - Если отклонение **меньше порога предупреждения** – состояние `NORMAL`.
   - Если отклонение **между порогом предупреждения и аварийным порогом** – может публиковаться `DEVIATION_WARNING`.
   - Если отклонение **превышает аварийный порог**:
     - публикуется событие `EMERGENCY_LAND_REQUIRED`;
     - состояние становится `EMERGENCY`.

4. **Взаимодействие с другими компонентами**
   - Компонент `emergensy` получает событие как `limiter_event` на своём топике `agrodron.emergensy` (доставляется через МБ).
   - Автопилот может использовать эти события только как информационные (логика аварийной посадки у вас уже вынесена в другие модули).

---

## 5. Параметры `.env` (основные)

См. `components/limiter/.env.example`. Ключевые параметры:

- интервалы polling и control-loop
- таймаут `LIMITER_REQUEST_TIMEOUT_S`
- пороги `LIMITER_MAX_DISTANCE_FROM_PATH_M`, `LIMITER_MAX_ALT_DEVIATION_M`
