# Сценарии сквозных тестов модели 

Ниже — 3 ключевых сквозных (end-to-end) сценария испытаний всей экосистемы «заказчик → агрегатор → эксплуатант → дрон/наземная станция/дронопорт → ОрВД БВС → страховая → регулятор», с критериями успешности. Формулировки целей безопасности (ЦБ) даны как подлежащие фиксации в сертификатах регулятора в терминах ГОСТ 72118-2025: т.е. в виде проверяемых целей по предотвращению/ограничению ущерба, переводу в безопасное состояние при отказах, управлению рисками и обеспечению доказуемости (трассируемости) выполнения.

## Общие предпосылки для всех 3 сценариев (единые требования испытаний)

1. У всех участников есть действующие сертификаты регулятора (агрегатор, эксплуатант, тип/модель дрона, наземная станция, дронопорт; ОрВД БВС).
2. В сертификатах явно зафиксированы ЦБ (идентификаторы, область действия, допущения, метрики/пороги, требования к журналированию).
3. Включено сквозное журналирование и неизменяемость ключевых событий (аудит): идентификаторы заказа/миссии/дрона/пилота-оператора/страхового покрытия/разрешений ОрВД БВС.


## Сценарий 1. «Нормальная миссия в регулируемом воздушном пространстве»: заказ → план полёта → запуск из дронопорта → выполнение → возврат/завершение → закрытие заказа и страхового покрытия

### Цель сценария

Проверить, что типовой бизнес-процесс выполняется без нарушений ЦБ и регуляторных ограничений, с подтверждением разрешений, валидностью сертификатов и корректной координацией с организатором воздушного движения БВС.

### Покрываемые цели безопасности (примерно-типовые ЦБ для фиксации в сертификатах)

* ЦБ-1 (предотвращение вреда третьим лицам): при любых обстоятельствах перемещения осуществляются в рамках заданных в аутентичном задании ограничений
* ЦБ-2 (управляемость и безопасное выполнение): при любых обстоятельствах выполняются только авторизованные ОрВД БВС перемещения
* ЦБ-3 (достоверность разрешений/сертификаций): при любых обстоятельствах выполняются полёты только с действительными сертификатами и страховым полисом
* ЦБ-4 (трассируемость/доказуемость): при любых обстоятельствах события безопасности записываются в защищённый бортовой журнал, только авторизованные пользователи имеют доступ к конфиденциальной информации
* ЦБ-5 (безопасность управления): при любых обстоятельствах выполняются только авторизованные критичные команды

### Участники/подсистемы

Агрегатор, эксплуатант, НУС (наземная управляющая станция), дрон (например, доставщик или инспектор), дронопорт, ОрВД БВС, регулятор (проверка валидности), страховая (активация покрытия на миссию).

### Шаги испытания (кратко)

1. Клиент оформляет заказ с параметрами (маршрут/зона, масса, время, требования).
2. Агрегатор выбирает эксплуатанта и дрон, проверяет сертификаты/ограничения, формирует миссию.
3. Запрос разрешения в ОрВД БВС (план полёта, геозоны, окна времени).
4. Страховая подтверждает покрытие на миссию (условия/лимиты/франшиза/исключения).
5. Подготовка в дронопорту, предполётные проверки, запуск.
6. Полёт с телеметрией, мониторингом ограничений, завершение миссии.
7. Закрытие заказа, автоматическое формирование отчётных пакетов (журналы, трек, события), завершение/архивация страхового покрытия.

### Критерии успешности (pass/fail)

1. Валидность допуска к миссии: запуск невозможен, если хотя бы один из атрибутов невалиден: сертификат дрона/эксплуатанта/агрегатора/НУС, разрешение ОрВД БВС, страховое покрытие.
2. Соблюдение разрешённых ограничений: по данным трека отсутствуют выходы за геозоны/высотные эшелоны/временные окна, указанные ОрВД БВС.
3. Соблюдение минимальной эшелонизации/разведения: не фиксируются события «опасного сближения» (c границами зоны безопасности) (по критериям, указанным в сертификате/правилах ОрВД БВС).
4. Непрерывность наблюдаемости: телеметрия и статус миссии доступны агрегатору и эксплуатанту с допустимыми потерями/задержками не хуже порогов, указанных в ЦБ; при деградации — корректная сигнализация.
5. Полнота доказательной базы: сформирован и сохранён пакет доказательств (журналы агрегатора/НУС/дрона/дронопорта/ОрВД БВС, хэш/подписи при применимости), позволяющий восстановить цепочку решений «заказ → допуск → полёт → завершение».
6. Корректное закрытие финансово-страхового контура: заказ закрыт, факт выполнения подтверждён, страховое покрытие переведено в завершённое состояние, отчёт доступен.


## Сценарий 2. «Отказ/инцидент в полёте»: потеря C2/навигации/нештатное препятствие → аварийная процедура → уведомления → страховой кейс → расследование и регуляторная отчётность

### Цель сценария

Проверить, что при отказе/атаке/сбое система:
* обнаруживает опасное отклонение,
* переводит дрон в безопасное состояние (contingency),
* координируется с ОрВД БВС,
* правильно запускает страховой и регуляторный процессы,
* сохраняет доказательства для расследования.

### Покрываемые цели безопасности (типовые)

* ЦБ-2 (управляемость и безопасное выполнение + fail-safe / safe-state) ==> при потере управления/позиционирования — переход в предопределённое безопасное состояние (возврат, зависание с ограничением, посадка в безопасной зоне, аварийная посадка) согласно сертифицированной логике.
* ЦБ-2 (управляемость и безопасное выполнение + fail-safe / safe-state) ==> ограничить время от выявления отказа до запуска аварийной процедуры.
* ЦБ-1 (предотвращение вреда третьим лицам) ==> исключить неконтролируемый полёт; исключить вход в запрещённые зоны; минимизировать кинетический риск.
* ЦБ-4 (трассируемость/доказуемость) ==> неизменяемые журналы, корректная классификация инцидента, уведомления заинтересованных сторон, формирование страхового события.

### Инъекции (варианты для теста)

* Потеря канала управления (C2) на заданное время.
* Деградация/подмена GNSS (спуфинг/джамминг) или резкое расхождение навигационных оценок.
* Внезапное ограничение ОрВД БВС (dynamic geofence) на маршруте.
* Отказ критического узла (например, батарея/мотор — в рамках допуска к испытаниям).

### Критерии успешности (pass/fail)

1. Детектирование: система фиксирует событие отказа/аномалии и классифицирует его (тип, критичность) в течение времени T_(detect), указанного в ЦБ/сертификате.
2. Переход в безопасное состояние: в течение T_(safe) после детектирования дрон выполняет сертифицированную аварийную процедуру; запрещённые состояния (например, продолжение миссии при потерянном C2 сверх допуска) не допускаются.
3. Соблюдение ограничений ОрВД БВС даже в аварии: дрон не пересекает запретные зоны; при введении dynamic geofence — маршрут/режим адаптируется либо выполняется посадка/возврат в допустимой области.
4. Оповещение:
   – эксплуатант и агрегатор получают уведомление об инциденте в течение T_(notify);
   – ОрВД БВС получает статус/намерения (contingency intent) в установленном протоколом формате;
   – при критериях страхового случая — автоматически создаётся страховое событие.
5. Страховой процесс: корректно фиксируются: время/место/причина (предварительно), идентификатор миссии, предполагаемый ущерб, перечень затронутых сторон; соблюдается согласованность данных между агрегатором, эксплуатантом и страховой.
6. Доказательная база для расследования: сохранены журналы «до/во время/после» инцидента, включая команды НУС, телеметрию, решения автопилота, сообщения ОрВД БВС; обеспечена целостность (контрольные суммы/подписи — если предусмотрено ЦБ).
7. Отсутствие вторичных нарушений безопасности: не возникает «цепочки отказов» из-за ошибок интеграции (например, повторный автоматический старт миссии, некорректная переинициализация, потеря идентификаторов).


## Сценарий 3. «Комплексная многосторонняя операция в плотной среде»: несколько эксплуатантов + разные типы дронов + дрон-матка + дронопорт + проверка сертификатов/отзывов (revocation) + конфликт трафика

### Цель сценария

Проверить устойчивость системы в условиях:
* одновременных миссий разных эксплуатантов,
* разнородного флота (доставщик/инспектор/агродрон/дрон-матка),
* высокой нагрузки на агрегатор/ОрВД БВС/дронопорт,
* изменения регуляторного статуса (например, отзыв/приостановка сертификата),
* конфликтов по трафику и ресурсам (слоты дронопорта, окна ОрВД БВС).

### Покрываемые цели безопасности (типовые)

* ЦБ-1-3 (недопущение эксплуатации без действующих допусков): мгновенная блокировка назначения/старта/продолжения миссии при отзыве/истечении сертификата или несоответствии конфигурации.
* ЦБ-1-3 (управление плотностью и конфликтами): предотвращение конфликтов планов/маршрутов/слотов; безопасная деградация при перегрузке.
* ЦБ-1-3 (сегрегация ответственности и данных): корректное разграничение ролей, исключение «переприсвоения» миссий/дронов, целостность данных между организациями.
* ЦБ-5 (безопасность составных систем, включая “дрон-матку”): безопасные процедуры запуска/возврата дочерних дронов, исключение опасных режимов при стыковке/расстыковке.

### Конфигурация теста

* Минимум 2 эксплуатанта, 1 агрегатор, 1 дронопорт, 1 ОрВД БВС.
* В воздухе/в подготовке одновременно несколько миссий разных типов.
* В середине теста регулятор публикует событие: отзыв/приостановка сертификата одного типа дрона или НУС, либо изменение условий эксплуатации (например, запрет полётов в зоне).

### Критерии успешности (pass/fail)

1. Проверка сертификатов и отзывов в реальном времени:
   – агрегатор не назначает миссию ресурсам с недействительным сертификатом;
   – если отзыв произошёл после назначения, но до старта — старт блокируется;
   – если отзыв произошёл в полёте и правила требуют прерывания — инициируется предписанная процедура (возврат/посадка) в рамках ЦБ.
2. Конфликт-менеджмент планов и ресурсов:
   – ОрВД БВС не подтверждает конфликтующие планы; агрегатор корректно перепланирует;
   – дронопорт корректно управляет слотами (зарядка/погрузка/взлёт/посадка), не допуская опасных пересечений операций.
3. Стабильность при нагрузке:
   – при пиковых событиях (массовые обновления статусов, перепланирование) система не теряет консистентность: нет «двойных назначений» одного дрона, нет «потерянных миссий», нет рассинхронизации статусов между агрегатором и эксплуатантом.
4. Безопасность операций “дрон-матка”:
   – расстыковка/стыковка дочерних дронов проходит только в разрешённых зонах/режимах;
   – при отказе одного дочернего дрона срабатывают локальные contingency-процедуры без каскадного нарушения безопасности остальных.
5. Межорганизационная трассируемость:
   – по каждому событию (назначение, старт, разрешение ОрВД БВС, отзыв сертификата, перепланирование, завершение) существует единая цепочка идентификаторов и журналов у всех сторон, пригодная для регуляторной проверки.
6. Отсутствие “теневой эксплуатации”:
   – невозможно обойти агрегатор/регуляторные проверки через прямое управление НУС (это либо технически запрещено, либо допускается только в режимах, явно описанных в сертификате и отражённых в журналах).

## Матрица трассировки

### Матрицы трассировки (сквозные сценарии → ЦБ → участники → проверки → артефакты)

Ниже — практичная «E2E-трассировка»: для каждого сценария указаны цели безопасности (ЦБ), кто их обеспечивает/проверяет, и какие артефакты требуются как доказательства.

#### Легенда артефактов (единообразные обозначения)

* AGG_EVT — журнал событий агрегатора (заказ/назначение/статусы/решения).
* OPS_EVT — журнал событий эксплуатанта (операционные решения/диспетчеризация).
* GCS_CMD — журнал команд НУС (наземной управляющей станции).
* UAS_TLM — телеметрия дрона (позиция/скорость/высота/режимы/ошибки).
* UAS_HLTH — журнал здоровья/диагностики/отказов (бортовые события).
* PORT_EVT — журнал дронопорта (доступ/слоты/заряд/погрузка/взлёт/посадка).
* ОрВД БВС_MSG — сообщения с ОрВД БВС (заявки/разрешения/dynamic geofence/конфликты).
* REG_STATUS — подтверждение статусов сертификатов/отзывов (ответы регулятора или доверенного реестра).
* INS_POL — страховой полис/покрытие (идентификатор, условия, период).
* INS_CLM — страховой кейс/уведомления/пакет заявления.
* AUD_HASH — контроль целостности (хэши/подписи/цепочки), если предусмотрено ЦБ.
* NET_PCAP — сетевые дампы (при тестировании протоколов/времени доставки/инцидентов).
